## 1. Purpose and Scope

This document specifies every external API endpoint of Themis Machina: request schemas, response schemas, authentication requirements, rate limits, error codes, and streaming behavior.

It is the contract between the backend and the frontend, and between the backend and any future integrations (MCP servers, partner applications, developer APIs).

The API is a **FastAPI** application exposing REST/JSON endpoints and Server-Sent Events (SSE) for streaming. All endpoints are versioned under `/api/v1/`. The OpenAPI specification is auto-generated by FastAPI and available at `/api/docs` in development and staging environments.

---

## 2. API Design Principles

1. **Versioned from day one.** All endpoints are under `/api/v1/`. Breaking changes go to `/api/v2/`. No unversioned endpoints except health checks.
2. **JWT-first authentication.** Every authenticated endpoint requires a Bearer token. Anonymous endpoints accept an anonymous session token.
3. **Consistent error format.** All errors return the same shape with a machine-readable `code` and a human-readable `message`.
4. **Streaming over polling.** Long-running operations (generation, document ingestion) use SSE, not polling.
5. **Explicit pagination.** All list endpoints use cursor-based pagination.
6. **Strict schemas.** All request bodies are validated with Pydantic; all response bodies are typed. FastAPI enforces this automatically.

## 3. Authentication

### 3.1 Authentication methods

| Method | Who | How |
|---|---|---|
| Bearer JWT | Authenticated users | `Authorization: Bearer <token>` header |
| Anonymous session token | Anonymous users | `X-Session-Token: <token>` header |
| Internal service token | Backend services | `X-Service-Key: <key>` header (not exposed externally) |

### 3.2 JWT structure

JWTs are issued by Clerk (Phase A) and signed with RS256. The backend verifies using Clerk's JWKS endpoint. Claims:

```json
{
  "sub": "user_01JXXXXXXXXXXXXXX",  // Clerk user ID
  "email": "user@example.com",
  "role": "public_user",            // themis_role claim
  "professional_verified": false,
  "iat": 1700000000,
  "exp": 1700003600,
  "iss": "https://clerk.themismachina.app"
}
```

The backend maps `sub` to an internal `user_id` UUID on first request (or looks it up on subsequent requests).

### 3.3 Anonymous session tokens

Anonymous sessions are granted by calling `POST /api/v1/sessions/anonymous`. The response contains a session token (opaque string) which is stored in a secure cookie (HttpOnly, SameSite=Strict). The token is validated against the `anonymous_sessions` table in Postgres.

### 3.4 Rate limiting (enforced at API gateway)

| User type | Queries/hour | Uploads/hour | Conversations/day |
|---|---|---|---|
| Anonymous | 30 | 0 | 5 |
| Public user (authenticated) | 200 | 5 | 50 |
| Professional user (verified) | 500 | 25 | unlimited |
| Admin | unlimited | unlimited | unlimited |

Rate limit headers returned on every response:

```
X-RateLimit-Limit: 200
X-RateLimit-Remaining: 187
X-RateLimit-Reset: 1700003600
```

### 3.5 Error responses

All errors follow a consistent envelope:

```json
{
  "error": {
    "code": "RATE_LIMIT_EXCEEDED",
    "message": "You have exceeded the 200 queries/hour limit. Resets in 47 minutes.",
    "details": {
      "limit": 200,
      "reset_at": "2024-12-01T15:00:00Z"
    }
  }
}
```

Standard error codes:

| HTTP status | Code | When |
|---|---|---|
| 400 | `VALIDATION_ERROR` | Request body fails schema validation |
| 401 | `UNAUTHORIZED` | Missing or invalid auth token |
| 403 | `FORBIDDEN` | Valid token but insufficient permissions |
| 404 | `NOT_FOUND` | Resource does not exist or not accessible to this user |
| 409 | `CONFLICT` | Duplicate resource (e.g., creating a matter with duplicate name) |
| 422 | `UNPROCESSABLE` | Request is valid but cannot be processed (e.g., unsupported file type) |
| 429 | `RATE_LIMIT_EXCEEDED` | Too many requests |
| 500 | `INTERNAL_ERROR` | Unexpected server error |
| 503 | `SERVICE_UNAVAILABLE` | LLM provider down, fallback also unavailable |

## 4. Shared Data Models

These Pydantic models are referenced throughout the endpoint specs.

### 4.1 Pagination

```python
class CursorPage(BaseModel, Generic[T]):
    items: list[T]
    next_cursor: str | None  # None if no more pages
    total_count: int | None  # None when count is expensive
```

### 4.2 Citation

```python
class Citation(BaseModel):
    source_id: str          # chunk ID in the corpus
    source_type: str        # "statute", "case_law", "patent", "user_doc", "web"
    tier: int               # 1, 2, 3, or 4
    display_ref: str        # e.g., "[s1]"
    locator: str | None     # e.g., "para 26", "Section 420"
    is_quote: bool
    # Source metadata (denormalized for display)
    source_title: str
    source_citation: str | None  # formal citation string
    source_date: str | None
    court: str | None
    excerpt: str            # the relevant excerpt (max 300 chars)
    archived_url: str | None  # for web sources
```

### 4.3 Message (in responses)

```python
class MessageResponse(BaseModel):
    id: str
    conversation_id: str
    turn_index: int
    role: str  # "user" | "assistant"
    content: str  # plain text of the message
    # For assistant messages only:
    claims: list[ClaimResponse] | None
    caveats: list[str] | None
    follow_up_suggestions: list[str] | None
    citations: list[Citation] | None
    confidence: str | None  # "high" | "medium" | "low"
    verification_status: str | None  # "verified" | "partial" | "unverified"
    model_version: str | None
    prompt_version: str | None
    latency_ms: int | None
    created_at: str
```

class SourceChunk(BaseModel):

### 5.4 Chat (message sending with SSE streaming)

This is the core endpoint. It accepts a user message and streams the response.

#### `POST /api/v1/conversations/{conversation_id}/messages`

Send a message and receive a streamed response.

**Auth:** JWT or anonymous session

**Request:**

```json
{
  "content": "Has the SC overruled Essar Steel post-2022 on equitable distribution?",
  "mode_override": null  // optional: override the conversation's mode for this turn
}
```

**Response:** `Content-Type: text/event-stream`

The response is a stream of Server-Sent Events. Event types:

**`retrieval_start`** — retrieval has begun:

```
event: retrieval_start
data: {"status": "searching", "corpora": ["case_law", "statute"]}
```

**`retrieval_complete`** — retrieval done, generation starting:

```
event: retrieval_complete
data: {"chunk_count": 8, "strategy": "hybrid_rerank"}
```

**`text_delta`** — a token or small group of tokens from the LLM:

```
event: text_delta
data: {"delta": "No, *Essar Steel* has not been overruled."}
```

**`citation_found`** — LLM emitted a citation reference:

```
event: citation_found
data: {
  "ref": "s1",
  "source_id": "case_dbs_bank_2024",
  "source_type": "case_law",
  "tier": 1,
  "display_ref": "[s1]",
  "source_title": "DBS Bank Ltd. v. Ruchi Soya Industries",
  "source_citation": "(2024) ...",
  "excerpt": "The principle of equitable distribution...",
  "locator": "para 31"
}
```

**`verification_start`** — citation verification starting:

```
event: verification_start
data: {"claim_count": 4}
```

**`verification_update`** — a claim's verification result:

```
event: verification_update
data: {
  "claim_index": 0,
  "status": "supported",
  "citations_verified": ["s1"]
}
```

**`message_complete`** — the full message is done:

```
event: message_complete
data: {
  "message_id": "msg_01JXXXXXXXXXXXXXX",
  "turn_index": 1,
  "content": "No, Essar Steel has not been overruled...",
  "claims": [...],
  "caveats": ["NCLAT has taken a slightly different line..."],
  "follow_up_suggestions": [
    "Compare with NCLAT line",
    "What did Essar Steel itself hold?",
    "Recent HC applications"
  ],
  "verification_status": "verified",
  "model_version": "nvidia/meta/llama-3.3-70b-instruct",
  "prompt_version": "generate_case.v1.2",
  "latency_ms": 8234
}
```

**`error`** — something went wrong:

```
event: error
data: {
  "code": "LLM_UNAVAILABLE",
  "message": "Generation service is temporarily unavailable. Please try again.",
  "retry_after_ms": 3000
}
```

**`scope_refusal`** — query was outside scope:

```
event: scope_refusal
data: {
  "refusal_type": "advice_seeking",
  "message": "I can't tell you what to do — that's a question for a qualified advocate...",
  "alternatives": [
    "Explain the law that applies",
    "Show how courts have ruled in similar cases"
  ]
}
```

**`clarification_needed`** — the system needs more information:

```
event: clarification_needed
data: {
  "question": "When you say 'the Tata case', do you mean one of these?",
  "options": [
    {"id": "tata_iron_steel_1958", "label": "Tata Iron and Steel Co. v. State of Bihar (1958)"},
    {"id": "tata_cellular_1994", "label": "Tata Cellular v. Union of India (1994)"},
    {"id": "tcs_ap_2004", "label": "Tata Consultancy Services v. State of AP (2004)"}
  ]
}
```

### 5.7 Document upload endpoints

#### `POST /api/v1/documents/upload`

Initiate a document upload.

**Auth:** JWT (anonymous users cannot upload)

**Request:** `multipart/form-data`

| Field | Type | Required | Description |
|---|---|---|---|
| `file` | binary | yes | The document file |
| `conversation_id` | string | yes | Which conversation to associate with |
| `matter_id` | string | no | Optionally scope to a matter |
| `document_type` | string | no | "contract" / "notice" / "judgment" / "other" |
| `user_role` | string | no | For contracts: "buyer" / "seller" / "licensor" / etc. |

**Validation:**
- `file` must be PDF, DOCX, or image (PNG, JPG, TIFF)
- `file` must be under 25 MB
- User must have document upload quota remaining

**Response `202`:** accepted, processing asynchronously

```json
{
  "document_id": "doc_01JXXXXXXXXXXXXXX",
  "status": "parsing",
  "status_url": "/api/v1/documents/doc_01JXXXXXXXXXXXXXX/status"
}
```

**Response `200`:**

## 6. WebSocket Support

The API does not expose WebSockets in v1.0. SSE is preferred because:

- SSE is unidirectional (server → client), which matches the generation use case perfectly
- SSE works over standard HTTP/2; no upgrade handshake needed
- SSE is natively supported by the `EventSource` browser API
- SSE reconnects automatically on disconnect

If bidirectional communication is needed in future (e.g., for voice streaming), WebSockets will be added in v1.1.

## 8. API Client SDK

For the frontend's use of the API, a typed TypeScript client is generated from the OpenAPI spec:

```bash
npx openapi-typescript /api/openapi.json --output src/lib/api-types.ts
```

This ensures the frontend TypeScript types always match the backend schemas, and compile-time errors catch mismatches before runtime.

For SSE consumption, a typed client wrapper handles event parsing:

```typescript
export class ThemisSSEClient {
  async sendMessage(
    conversationId: string,
    content: string,
    onEvent: (event: SSEEvent) => void
  ): Promise<void> {
    const response = await fetch(`/api/v1/conversations/${conversationId}/messages`, {
      method: 'POST',
      headers: { 'Content-Type': 'application/json', 'Authorization': `Bearer ${token}` },
      body: JSON.stringify({ content }),
    });

    const reader = response.body!.getReader();
    const decoder = new TextDecoder();
    let buffer = '';

    while (true) {
      const { done, value } = await reader.read();
      if (done) break;
      buffer += decoder.decode(value, { stream: true });
      const events = parseSSEBuffer(buffer);
      for (const event of events) {
        onEvent(event);
      }
    }
  }
}
```

## 9. API Versioning Policy

- **v1.0** — current version, documented in this document
- **Breaking changes** (removing fields, changing field types, removing endpoints) → new version `/api/v2/`
- **Additive changes** (new optional fields, new endpoints) → no version bump, backward compatible
- **Deprecation notice** — deprecated endpoints serve an `X-Deprecated: true` header and a `deprecation_message` in the response for at least 60 days before removal
